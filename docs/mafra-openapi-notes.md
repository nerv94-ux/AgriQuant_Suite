# 가락 OpenAPI · 품목 코드 — 팀 공용 메모

명세 원본은 사용자 제공 파일(다운로드 폴더 등)에 두고, **이 문서는 코드와의 대응·주의사항만** 적는다. 원본 파일명을 바꾸거나 버전이 바뀌면 여기 날짜를 갱신할 것.

## 1. 원본 자료 (참고 경로)

| 구분 | 예시 파일명 |
|------|-------------|
| 실시간 경락 정보 | `도매시장 실시간 경락 정보.xls` → API `Grid_20240625000000000654_1` |
| 정산 가격(원천) | `도매시장 원천데이터 정산 가격.xls` → 코드베이스 `Grid_20240625000000000653_1` |
| 농수축산물 도매가격 | `농수축산물 도매가격.xls` (별도 그리드 — 코드에서 사용하는 ID와 대조) |
| 품목 코드 | `품목 코드.xls` → `Grid_20240626000000000668_1` |
| 코드 계층(표) | `농수축산물+도매가격+관련+코드.csv` |

권장: 위 파일을 `docs/attachments/mafra/` 아래에 **복사본**을 두면 이후 세션에서도 동일하게 참고 가능.

## 2. 실시간 경락 API — 입력 파라미터 (명세 xls 기준)

샘플 URL 패턴: `http://211.237.50.150:7080/openapi/.../Grid_20240625000000000654_1/...`

| 파라미터 | 설명(요지) | 크기(명세) | 코드베이스 |
|----------|------------|------------|------------|
| SALEDATE | 경매일자 | 8 (YYYYMMDD) | ✅ |
| WHSALCD | 도매시장코드 | 6 | ✅ |
| CMPCD | 법인코드 | 8 | 선택 |
| LARGE | 대분류코드 | **2** | `mafraLarge` |
| MID | 중분류코드 | **2** | `mafraMid` |
| SMALL | 소분류코드 | (명세 원본에서 길이 확인 — 데스크는 **응답 행 필터·조회의 기준**) | `mafraSmall` |

**주의:** 명세상 `LARGE`/`MID`는 **문자열 2자** 규격이다. 팀에서 말하는 “8자리 품종코드”와 **필드 분해 방식이 같은지**는 원본 명세·샘플 응답으로 한 번 더 맞춰야 한다. (아래 CSV 계층과 가락 그리드 컬럼이 1:1인지 검증 필요.)

## 3. 도매가격 관련 코드 CSV — 계층 예시

컬럼: `부류코드`, `부류명`, `품목코드`, `품목명`, `품종코드`, `품종명`

예: 당근 무세척 — `200`, 채소류 / `232`, 당근 / `01`, 무세척.

**가능한 8자리 조합(가설):** `부류(3) + 품목(3) + 품종(2)` → 예: `20023201`.  
가락 API의 `LARGE`/`MID`/`SMALL`이 이 세 덩어리와 **어떻게 대응하는지**는 반드시 **실제 API 응답 한 건**과 대조해 확정할 것.

## 4. 이 레포 코드와의 매핑

| 용도 | 파일 | 그리드 ID |
|------|------|-----------|
| 품목코드 동기화·검색 | `mafra-item-code/client.ts` | `Grid_20240626000000000668_1` |
| 실시간 경락 | `mafra-rltm-auc-info/client.ts` | `Grid_20240625000000000654_1` |
| 정산 가격 정보 | `mafra-clcln-prc-info/client.ts` | `Grid_20240625000000000653_1` |

데스크 시세표(`deskAuctionPriceSheet`)는 실시간 경락 호출 시 **시장별 대·중 불일치 완화를 위해 API에는 `SMALL`만 전달**하는 정책을 취할 수 있음(구현 시점의 주석 참고).

## 5. 다음에 확정하면 좋은 것 (체크리스트)

- [ ] 실시간 경락 명세 xls에서 **SMALL** 입력 필드의 **정확한 길이·필수 여부**
- [ ] 동일 품목에 대해 **품목코드 그리드**의 `LARGE`/`MID`/`SMALL` 한 줄과 **CSV의 부류·품목·품종** 숫자가 어떻게 합쳐지는지
- [ ] 응답 `row`의 `LARGE`/`MID`/`SMALL`이 입력과 동일 스키마인지

## 6. 코드 정책 (구현)

- `mafraSmallCodesMatch`: 응답 `SMALL`과 저장값은 **trim 후 완전 일치**만 (느슨한 패딩은 서로 다른 품목이 섞여 보이는 원인이 됨).
- `expectedSmall`이 비면 응답 행을 **통과시키지 않음**(`[]`).
- 데스크 `realtime-auction` / 전국 시세: DB에 **대·중·소가 모두** 있으면 API에도 **세그먼트를 함께** 넣고, 응답도 세 값으로 거른다. **`SMALL`만 짧은 숫자(예: 01)** 이면 품목이 유일하지 않을 수 있음.
- 코드사전 **데스크 전용** 품목 매칭(`pickItemCandidatesForDesk`): 예전에는 `SMALL || MID`로 **중분류 코드**가 소분류처럼 쓰여 경매가 넓게 터지는 경우가 있었음 → **`SMALL`만** 사용(비어 있으면 그 행 제외).
- 실시간 경락 JSON 파싱: `row`가 배열이 아니라 **객체 한 건**일 때도 처리; `SMALL`/`small` 등 키 변형 시도.
